# Инфоцентр РЦК — как это устроено

Дашборд для ЦПП Калининградской области (React + Vite + TypeScript + Recharts),
встроен в Битрикс24 как локальное приложение. Развёрнут в июле 2026.

## Что было сделано

**14.07.2026 — сборка, фикс бага, деплой.**

1. Собрали проект (`npm install && npm run build`), не трогая логику интеграции
   со Списками Битрикс24 (`src/App.tsx`, блок `LISTS`, ID 78–87) — она уже была
   проверена и работала.

2. **Починили баг с пустыми графиками.** На вкладках «Качество» и «Персонал»
   часть графиков не рисовалась (высота контейнера — 0, SVG отсутствовал).
   Причина — не анимация вкладок (AnimatePresence), а CSS: обёртки графиков
   имели класс `flex-1` внутри карточки `flex flex-col` без явной высоты
   у родителя. По спецификации flexbox в такой ситуации `height: 100%` у
   `ResponsiveContainer` схлопывается в 0. Убрали `flex-1` из трёх мест
   (`QualityPanel.tsx`, `PersonnelPanel.tsx`), оставив только `h-[...]`.
   Проверено headless-Chromium (Playwright) по всем 5 вкладкам — все 11
   графиков рендерятся.

3. Добавили `base: './'` в `vite.config.ts` — пути к `assets/index.js` и
   `assets/index.css` в `dist/index.html` стали относительными (без этого
   Vite ставит абсолютные `/assets/...`, и сборка не открывается не из
   корня домена).

## Хостинг: Cloudflare Workers (не Диск Б24, не GitHub Pages)

Перебрали три варианта, прежде чем остановились на рабочем:

- **Диск Битрикс24** — не подходит вообще. Облачный Б24 блокирует загрузку
  HTML-файлов со `<script>` («потенциально опасные конструкции»), а даже
  если бы пропустил — Диск не отдаёт файлы как веб-страницы, относительные
  пути между ними не работают.
- **GitHub Pages** — сборка открывается по прямой ссылке, но отвечает
  `405 Method Not Allowed` на POST. А Битрикс24 открывает локальное
  приложение именно POST-запросом (передаёт токен авторизации). Итог:
  приложение не открылось бы изнутри портала.
- **Cloudflare Workers** — сработал. Финальная схема:
  - `wrangler.jsonc` — конфиг: воркер `worker.js` + статика из `dist/`
    через биндинг `ASSETS`, с `"run_worker_first": true` (иначе Cloudflare
    сначала пытается отдать статику напрямую и всё равно падает на POST).
  - `worker.js` — при `POST` переигрывает запрос в `GET` и отдаёт тот же
    `index.html`; на `GET` отдаёт статику как есть.
  - Деплой: `npx wrangler deploy` (учётка Cloudflare — westwoodgpt@gmail.com).
  - Боевой URL: **https://infocenterrck.westwoodgpt.workers.dev/**

## Как дашборд получает данные из Битрикс24

Хостинг (Cloudflare) отдаёт только программу — три статических файла, без
доступа к каким-либо данным. Данные подключаются уже внутри портала:

1. Сотрудник открывает пункт меню → Битрикс24 грузит наш `index.html` в
   `iframe` и передаёт туда токен авторизации POST-запросом.
2. `index.html` условно подключает SDK Битрикс24
   (`api.bitrix24.com/api/v1/`) — **только если страница открыта внутри
   iframe** (`window.self !== window.top`), иначе при прямом заходе на
   Cloudflare-URL дашборд показывает «Автономный режим» с демо-данными
   из `src/data.ts`.
3. SDK создаёт `window.BX24`; код дашборда дёргает
   `BX24.callMethod('lists.element.get', ...)` — запрос уходит не на
   Cloudflare, а к родительскому окну (порталу), который сам обращается
   к своим Спискам и возвращает результат в iframe.

Данные Списков наружу (на Cloudflare/GitHub) никогда не отправляются —
только код туда-обратно между порталом и iframe в браузере сотрудника.

## Настройка в Битрикс24

Приложения → Разработчикам → Другое → Локальное приложение:
- Путь обработчика: `https://infocenterrck.westwoodgpt.workers.dev/`
- Права: `lists`
- «Использует только API» и BitrixMobile — выключены

В веб-версии портала пункт меню новых приложений уходит в конец списка
(«Ещё» внизу левого меню) — нужно закрепить через звёздочку «В избранное»
или администратору применить общее меню для всех сотрудников.
